# システム概要

進級制作にて制作した、飲食店の **注文〜調理〜配膳〜会計〜売上分析** 全体フローを支える**テーブルオーダーシステム**です。私たちグループ4は、**バーガーショップ**を想定したシステムを開発しました。

卓上タッチパネルでの注文から、キッチン・ホール・レジ各端末でのオペレーション、管理者による売上分析や端末管理まで、注文とその処理、分析に関係する部分をシステム化しています。

## 機能一覧

- **来店人数入力端末** — 店舗入口に設置。顧客が来店時に人数を入力する端末
  - 来店人数の入力
  - 顧客番号の発行
- **卓上端末** — 各テーブルに設置。顧客が注文・履歴確認・会計リクエストを行う端末
  - 顧客番号の入力
  - メニュー一覧・商品詳細の表示、オプション選択
  - 注文確定・キッチンへの送信
  - 注文履歴の確認
  - 会計リクエスト（未提供商品がある場合は警告表示）
- **キッチン端末** — キッチンスタッフが注文内容を確認し、調理状況を更新する端末
  - 新規注文の一覧表示
  - 調理開始操作（操作から5秒以内のキャンセル可）
  - 調理完了操作
- **ホール端末** — ホールスタッフが調理完了済みの注文を確認し、配膳を行う端末
  - 調理完了済み注文の一覧表示
  - 商品単位 / テーブル単位での配膳完了操作
- **レジ端末** — レジスタッフが顧客の注文内容を確認し、会計処理を行う端末
  - 顧客番号による注文内容・会計金額の確認
  - 会計確定操作
  - 未提供商品がある場合の会計中止 / 強制会計の選択
- **管理者端末** — 管理者がシステム全体を管理する端末
  - 管理者ログイン
  - 端末の登録・管理
  - 売上分析（日別・カテゴリ別など複数の切り口に対応）

# 振り返り

## 技術面

### 工夫点

- キッチン・ホール端末では、ユーザー操作を介さずにリアルタイム性を確保するため、5秒ごとにサーバーへ新規注文の有無を問い合わせるポーリング方式を採用した。
- 顧客番号や注文番号では、セキュリティと運用の両面で安全なID生成を行うため、予測不可能かつ重複しないIDであるUUIDを採用した。
- 卓上端末では、操作後の読み込みなどで発生する画面フリーズへの不安感を減らすため、Reactを採用して画面遷移や状態管理を効率化し、どうしても時間のかかる処理中はローディングスピナーの表示を行った。
- 端末識別では、顧客端末のなりすましや再起動の際のログイン操作を不要にするため、クライアント証明書を使用して同一端末として識別できるようにした。

### 苦労した点

**問題:** APIエンドポイントの実装において、各リソースからのレスポンス内容や形式などが統一できておらず、実装のたびにAPIの仕様を確認する必要があり、実装効率が大幅に低下していた。

**原因:** API設計に関するドキュメントが不十分かつ分散していたため、APIの仕様を一元的に把握できていなかったことが原因。

**解決:** API設計のドキュメントをOpenAPI形式で再整備し、今後はAPI仕様はこのドキュメントのみを正とすることをチーム内で合意した。これにより、APIの実装やフロントエンドからの利用の際にAPI仕様を一元的に把握できるようになり、実装効率が大幅に向上した。

### 今後の改善点

**セキュリティ**

- [ ] デバイス種別ごとにアクセス可能なページ / API を制限しセキュリティを強化
- [ ] DB 接続情報・ペッパー値の環境変数化
- [ ] 全コントローラの認可強化（端末識別・種別登録は実装済だが、種別ごとのアクセス制御が未対応のエンドポイントが多い）
- [ ] mTLS 終端を Tomcat から Apache HTTP Server へ移行

**UI / UX**

- [ ] 卓上画面の広告バナー画像がハードコーディングされているため、管理者端末での画像アップロード機能を追加して複数画像を動的に変更できるようにする
- [ ] デバイス登録時に同一テーブルへの重複設定がある場合の警告・ハイライト表示

**保守性**

- [ ] テストコードの再整備
- [ ] 顧客 ID 生成方式の UUID への完全移行（現状は暫定的に数値を UUID として使用）
- [ ] ログライブラリへの移行（`printStackTrace()` / `System.out.println()` を置き換え）

## チームワーク面

### 良かった点

- Notionのタイムラインビューをガントチャートとして活用し、適切な粒度に分割したタスクを管理することでスケジュール管理ができた
- 相互コードレビューにより、バグ・技術的負債の抑制と設計意図の共有ができた。メンバーにとっても実装パターンを習得する機会になった
- タスク遅延の早期発見と巻き取りにより、スケジュールの帳尻を合わせることができた
- 学校での作業時は直接話しかけ、それ以外はチャットで進捗報告・確認を行い、状況に応じたコミュニケーションが取れていた

### 改善点

- 報告タイミングが定まっておらずメンバーからの自発的な報告が少なかったため、定期的な進捗報告のタイミングを設けることでチーム規模が大きくなっても機能する運用にする
- 機能仕様のみで実装面の方針が曖昧だったため、実装アプローチも事前にドキュメント化することで各自の判断による実装のばらつきを減らす
- スケジュール優先で定型タスクに偏らせてしまったため、難易度の高いタスクも適切に割り当てることでメンバーの技術習得の機会を確保する

# 実行方法

`docker-compose.yml` を利用して、以下の手順でシステムを起動できます。ただし、事前に**Oracle Database 19c**のDockerイメージをビルドしておく必要があります。

## Oracle Database Dockerイメージのビルド

Oracleの公式Dockerイメージのリポジトリからスクリプトを利用して`oracle/database:19.3.0-ee`イメージをビルドします。以下の手順でビルドしてください。

```bash
#  1. Clone Git repository
git clone https://github.com/oracle/docker-images
cd docker-images/OracleDatabase/SingleInstance

#  2. `./README.md`に従いDBのバイナリをzip形式でダウンロードし配置。ARM版も利用可能。
#      配置先: docker-images/OracleDatabase/SingleInstance/dockerfiles/19.3.0
cp /tmp/LINUX.X64_193000_db_home.zip ./dockerfiles/19.3.0

# 3. スクリプトを利用してdockerイメージをビルド
./dockerfiles/buildContainerImage.sh -v 19.3.0 -e

# 完了。`docker image ls`で確認可能。
cd ../../../
# docker image ls
```

## システムの起動(APサーバー/DBサーバー)

```bash
# 1. このリポジトリをクローン
git clone https://github.com/itbsr/G04_Promotional-Production.git
cd G04_Promotional-Production

# 2. docker-composeでシステムを起動(初回はDBのセットアップに時間がかかるため5~10分程度かかります)
docker compose up --build -d
```

# 注意点

- 顧客番号は授業内で必要なデバイスを用意できないため、0~999999の範囲でランダムに整数が生成される仕様になっています。実際の運用では、顧客番号発行時に顧客番号を含むQRコードを伝票として発行し各端末でQRコードを読み取る仕様を想定しています。
- 端末識別は授業内での課題提出の都合上、クライアント証明書だけでなく`device_id` クエリパラメータでの識別も行う仕様になっています。実際の運用では、クライアント証明書のみでの識別を想定しています。
- `device_id` クエリパラメータでアクセスする場合、異なる端末URL間でCookieが共有されると正常に動作しません。複数の端末URLを同時に確認する場合は、ブラウザのプロファイル機能やプライベートウィンドウを使用して、端末ごとにCookieが分離された状態でアクセスしてください。
